# Claude Desktop CN Native Gap Roadmap

这份路线图只关注一件事：和原生 Claude / Claude Code 相比，我们现在还差什么，下一刀最值得补什么。

## 已经落地的部分

最近几轮已经完成的关键工作：

- GitHub OAuth 已切换到用户自有 App
- GitHub 仓库导入可用
- 中文界面基础完成，并支持中英文 UI 切换
- `Code` 页已经具备真实工作区能力
  - 选择工作区
  - 文件树浏览
  - 预览与编辑
  - HTML 预览
  - 新建 / 删除 / 重命名
  - Git 状态与单文件 diff
  - 暂存 / 取消暂存 / 恢复
  - 提交 / 推送
- `Settings` 页已从纯骨架升级为可用设置框架
- `Cowork` 页已从空白占位升级为协作总览页
- Windows 关闭窗口后已支持转入系统托盘驻留
- 已支持 Skill 导入

## 当前最大的差距

现在真正的差距已经不再是“有没有页面”，而是“这些页面是不是已经形成了完整闭环”。

核心缺口主要有三类：

1. 协作闭环还不完整
2. 项目级规则 / 记忆 / 自动化还不够深
3. 与原生 Claude Code 相比，工作区行为和设置体系还不够成熟

## Phase 1：把现有入口做成完整闭环

### P0

1. Cowork 变成真正的协作区
   - 任务列表
   - 状态流转
   - 审阅记录
   - 和项目 / 聊天 / 代码页之间互跳

2. Settings 去骨架化
   - Git 默认行为补全
   - MCP 管理增强
   - 环境与工作树参数补全
   - 已归档与历史会话体验补全

3. Code 页面继续工程化
   - 更稳定的预览体验
   - 更清晰的文件操作反馈
   - 更完整的 Git 工作流

### P1

4. 项目规则与工作记忆
   - 项目级说明
   - 类似 `CLAUDE.md` 的入口
   - 项目记忆与知识关联

5. 更贴近原生的交互整理
   - 更紧凑的排版
   - 更清楚的状态提示
   - 更少“只能显示不能操作”的按钮

## Phase 2：继续向 Claude Code 靠拢

### P0

1. 工作区与权限体系继续下沉
   - 权限策略更明确
   - 工作区默认行为更完整
   - Shell / 编辑器 / Git 偏好更可控

2. 项目与代码联动
   - 项目直接打开工作区
   - 项目规则透传到代码工作流
   - 项目级 GitHub 源与本地文件联动

3. 自动化与扩展能力
   - MCP 管理增强
   - Hook / 自动化入口
   - 更明确的 Skill 管理体验

### P1

4. 更强的工程协作
   - 审阅面板
   - 任务执行历史
   - 项目状态总览

5. 发布与维护体验
   - 更顺滑的版本发布
   - 更清晰的更新说明
   - 更稳定的安装与升级链路

## 当前建议继续推进的主线

接下来最值得继续做的是：

1. 协作页继续做实
2. 设置页剩余区域补齐
3. Code 页再补一轮稳定性与体验
4. 文档、发布说明和版本流程标准化
